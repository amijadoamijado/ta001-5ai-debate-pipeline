# 要件定義書: 5AIエージェント議論パイプライン

## プロジェクト概要

5つの異なるAI（Claude・ChatGPT・Gemini・Grok・Perplexity）をエージェントとして連携させ、
経営意思決定を支援する「一人経営企画室」パイプライン。

- **対象組織**: 税理士法人スリーエス
- **原典**: `4ai_pipeline_requirements_v22.docx` (v2.2)
- **構築方針**: sd003フレームワークのAI協調ワークフローを参考パターンとして、ta001専用に新規構築
- **参考実装**: sd003 `/workflow:*` コマンド群、テンプレート、ハンドオフプロトコル

## 設計原則

1. **異種モデル間議論**: 学習データと推論特性が異なる5モデルで本質的な対立構造を作る
2. **5AI並行Deep Research**: 各AIの情報アクセス経路の違いを専門性として活用し、死角を消す
3. **Chrome自動化**: Claude in Chromeで各AIのWeb版Deep Researchをブラウザ操作（サブスク内）
4. **知的誠実性**: 全エージェントに「同意の前に弱点特定」「迎合は合意ではない」を最上位規範として適用
5. **ソース検証の独立**: Perplexityによる引用付きファクトチェックで、他4AIの主張を独立検証
6. **フレームワーク最小主義**: 案件タイプ別に「必須」と「任意」を分け、儀式化を防ぐ
7. **歴史的検証**: 「愚者は経験に学び、賢者は歴史に学ぶ」（ビスマルク）。Claude Codeが歴史家として、古今東西の成功・失敗・回復事例から現在の提案を検証する

### REQ-001.5: Phase 3.5 - 歴史的検証（Claude Code 歴史参謀）

> 「愚者は経験に学び、賢者は歴史に学ぶ」— オットー・フォン・ビスマルク

- **When**: Phase 3批判レポートが完成した時
- **Who**: Claude Code（ローカル実行・追加コストなし）
- **What**: 提案されている戦略・施策を、古今東西の歴史的事例と照合し、歴史家の視点から検証する
- **入力**: Phase 1提案書 + Phase 2補強レポート + Phase 3批判レポート
- **出力**: 歴史的検証レポート（構造化JSON + 人間可読マークダウン）
- **優先度**: P0（必須）
- **Claude Codeの歴史参謀としての役割**:

| 検証観点 | 具体的な分析 |
|---------|------------|
| **大成した企業・戦略** | 提案と類似の戦略で成功した歴史的事例。何が成功要因だったか |
| **失敗した企業・戦略** | 提案と類似の戦略で失敗した事例。何が致命的だったか |
| **回復した企業** | 一度失敗したが復活した事例。転換点は何だったか |
| **古今東西の戦略・戦術** | 孫子の兵法、ランチェスター戦略、ブルーオーシャン、**ポーターの競争戦略**（Five Forces + コストリーダーシップ/価格戦略、差別化戦略、集中/ニッチ戦略）、**アンゾフの成長マトリックス**（市場浸透・市場開拓・製品開発・多角化）、**BCGプロダクトポートフォリオマネジメント**（花形・金のなる木・問題児・負け犬）、**ヘンリー・フォードの改善**（大量生産・流れ作業・垂直統合・「顧客が望む色は黒」の標準化哲学）、**トヨタの改善（TPS）**（カイゼン・ジャストインタイム・自働化・ムダ排除・現地現物・なぜなぜ5回）、**イーロン・マスクの改善**（第一原理思考・高速イテレーション・垂直統合・不可能を疑え・物理法則以外は変更可能）等、提案に適用可能な歴史的戦略理論・改善哲学 |
| **時代の転換点との類似性** | 現在の状況が過去のどの転換点に類似するか（産業革命、IT革命、規制変更等） |
| **歴史が示す落とし穴** | 「歴史は繰り返す」パターン。提案が陥りやすい歴史的な罠 |

- **出力フィールド（歴史的検証固有）**:

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `historical_parallels[]` | array | 提案と類似の歴史的事例（成功・失敗・回復別） |
| `success_patterns[]` | array | 成功事例から抽出したパターンと提案への適用可能性 |
| `failure_warnings[]` | array | 失敗事例から抽出した警告と提案のリスク対応 |
| `recovery_playbooks[]` | array | 回復事例から抽出した「もし失敗した場合」のプレイブック |
| `strategic_frameworks[]` | array | 適用可能な歴史的戦略理論・改善哲学（ポーター、アンゾフ、BCG、フォード、トヨタTPS、マスク、孫子、ドラッカー等） |
| `era_similarity` | object | 現在の状況と過去の時代転換点との類似度分析 |
| `history_verdict` | object | 歴史の観点からの総合判定（support/caution/against + 根拠） |

- **リサーチツール活用**: Claude Codeは訓練データの知識に加え、以下のAPIで歴史的事例の裏付けを取る:
  - **Brave Search API**: プライバシー重視のWeb検索。企業の歴史、倒産事例、M&A履歴の検索
  - **Tavily API**: LLMエージェント最適化検索。構造化された企業分析・事例研究の取得
  - **Exa API**: セマンティック検索。「この戦略と類似の歴史的事例」を意味ベースで発見
- **ツール使い分け**:

| ツール | 最適な用途 | 例 |
|--------|----------|-----|
| Brave | 一般的な企業史・倒産事例の広範検索 | 「同業種 倒産 原因 事例」 |
| Tavily | 構造化されたビジネス分析の取得 | 企業の戦略転換事例のレポート |
| Exa | 類似パターンのセマンティック発見 | 「この戦略に似た歴史的成功/失敗」 |

- **歴史参謀の行動規範**:
  1. 「歴史は韻を踏む」— 完全な一致ではなくパターンの類似性に着目せよ
  2. 生存者バイアスを排除せよ — 成功事例だけでなく、同時期に同じ戦略で消えた企業も分析
  3. 時代背景の差異を明示せよ — 歴史的事例と現在の環境差を必ず注記
  4. 「この戦略は歴史上、何回試みられ、何回成功したか」を定量化せよ
  5. 訓練データの知識だけに頼るな — Brave/Tavily/Exaで最新の事例・データを裏取りせよ

- **受入基準**:
  - [ ] 成功事例が3件以上提示されていること
  - [ ] 失敗事例が3件以上提示されていること
  - [ ] 各事例に時代・地域・業種・結果・教訓が含まれること
  - [ ] 提案への具体的な示唆（Do/Don't）が導出されていること
  - [ ] 歴史的戦略理論が少なくとも3つ適用されていること（ポーター/アンゾフ/BCGを優先的に検討）
  - [ ] history_verdictに明確な根拠があること
  - [ ] 知的誠実性フィールドが出力されること

## REQ-004: 知的誠実性ルール（全エージェント共通）

### REQ-004.1: 4原則

Phase 0〜4の全エージェントに適用される最上位の行動規範。
全プロンプトテンプレートの冒頭に組み込む。

| # | 原則 | 具体的な行動規範 |
|---|------|----------------|
| 1 | 同意の前に弱点を特定せよ | 前のエージェントの主張に同意する前に、最も弱い点を特定。弱点を1つも見つけられないなら自分の理解が浅い可能性を疑え |
| 2 | 「妥当」は結論であり出発点ではない | 検証なしの「妥当」は出力禁止。「妥当である」は検証プロセスの結果としてのみ許される |
| 3 | 迎合は合意ではない | 前のエージェントの結論をそのまま追認することは「迎合」。独自検証を経ていない同意はパイプラインに価値を追加しない |
| 4 | 早すぎる収束は思考の放棄 | 議論の早期段階で結論に収束することを禁止。各フェーズは前フェーズの結論を「仮説」として扱い、反証可能性を常に保持 |

- **優先度**: P0（必須）

### REQ-004.2: 検証メカニズム（必須出力フィールド）

各フェーズの出力に以下のフィールドを必須化:

| フィールド | 内容 |
|-----------|------|
| `weakest_point_identified` | 前フェーズの主張の中で特定した最も弱い点 |
| `disagreements[]` | 前フェーズの結論と異なる見解リスト。ゼロの場合は理由を明記 |
| `verification_method` | 各claimへの同意・修正の判断に至った検証方法の説明 |

- **優先度**: P0（必須）
- **受入基準**:
  - [ ] Phase 1〜4の全出力にこれら3フィールドが含まれること
  - [ ] disagreementsが0件の場合に理由が明記されていること

## REQ-005: リサーチ統合ルール

### REQ-005.1: Perplexity信頼度調整

- `verified: true` → confidence引き上げ（+0.1〜+0.2）
- `verified: false` → confidence下方修正（-0.2〜-0.3）
- `verified: partial` → 注意フラグ付きで記録（confidenceは変更なし）
- **優先度**: P1（重要）

### REQ-005.2: 矛盾検出の強化

- Perplexityの`contradiction_flags`を最優先で処理
- ソース付き矛盾指摘は印象ベースの矛盾より信頼性が高い
- **優先度**: P1（重要）

### REQ-005.3: 技術実現性の反映

- Codexが「実現困難」と判断した施策はPhase 1でリスクとして明示する義務
- **優先度**: P1（重要）

### REQ-005.4: 統合レポート構成

```json
{
  "market_overview": {},
  "competitive_landscape": {},
  "customer_insights": {},
  "regulatory_environment": {},
  "social_intelligence": {},
  "source_verification": {},
  "technical_feasibility": {},
  "perception_vs_reality": [],
  "contradictions": [],
  "blind_spots": [],
  "research_quality": {}
}
```

- **優先度**: P1（重要）

### REQ-005.5: 感情データの位置づけ

- Grokの感情分析は「参考情報」
- 風評リスク「高」の場合のみ意思決定要因に格上げ
- **優先度**: P2（あると良い）

### REQ-005.6: 死角チェック

- 5AI+Codex全てが欠落した視点がないか確認
- `blind_spots[]`として統合レポートに記録
- **優先度**: P1（重要）

## REQ-010: リサーチツール統合（Brave / Tavily / Exa API）

### REQ-010.1: リサーチツール一覧と特性

Phase 0の5AI Deep Researchに加え、プログラマブルな検索APIを活用する。
これらのツールはPhase 0（補助検索）およびPhase 3.5（歴史的検証の裏取り）で使用される。

| ツール | API種別 | 特性 | 主な用途 |
|--------|---------|------|---------|
| **Brave Search API** | REST API | プライバシー重視、Googleと異なるインデックス、ニュース・Web両対応 | 企業史・倒産事例・規制動向の広範検索 |
| **Tavily API** | REST API | LLMエージェント最適化、構造化された回答、ソース付き | ビジネス分析・事例研究・定量データの構造化取得 |
| **Exa API** | REST API | セマンティック検索、類似コンテンツ発見、学術論文対応 | 類似戦略の意味ベース発見、学術的根拠の取得 |

- **優先度**: P1（重要）

### REQ-010.2: フェーズ別ツール活用マップ

| フェーズ | 使用ツール | 目的 |
|---------|-----------|------|
| **Phase 0** | Brave + Tavily | 5AI Deep Researchの補完。AIが見逃した情報の補助検索 |
| **Phase 2** | Tavily | Codex補強時の数値データ・統計の裏取り |
| **Phase 3** | Brave + Exa | Gemini批判時の反証事例・失敗事例の検索 |
| **Phase 3.5** | Brave + Tavily + Exa（全3ツール） | 歴史的検証の裏取り。訓練データの知識を最新情報で補強 |

- **優先度**: P1（重要）

### REQ-010.3: ツール使い分けルール

| 検索目的 | 最適ツール | 理由 |
|---------|-----------|------|
| 広範な事例検索（「○○業界 倒産 原因」） | Brave | Googleと異なるインデックスで死角を補完 |
| 構造化されたビジネスデータ取得 | Tavily | LLM最適化済み、そのままJSON化可能 |
| 「この戦略に似た事例」の発見 | Exa | 意味ベース検索で表層的なキーワード一致を超える |
| 最新ニュース・プレスリリース | Brave | ニュース検索モードで最新情報を取得 |
| 学術論文・研究データ | Exa | 学術コンテンツのセマンティック検索に強い |
| 定量データの裏取り（市場規模等） | Tavily | ソース付き構造化回答で信頼性を確保 |

- **優先度**: P1（重要）

### REQ-010.4: API設定と接続

- **環境変数**:

| 変数名 | 説明 |
|--------|------|
| `BRAVE_API_KEY` | Brave Search API キー |
| `TAVILY_API_KEY` | Tavily API キー |
| `EXA_API_KEY` | Exa API キー |

- **MCP対応**: 各ツールがMCPサーバーとして利用可能な場合はMCP経由で接続
- **フォールバック**: APIキー未設定時はスキップし、利用可能なツールのみで実行
- **優先度**: P1（重要）
- **受入基準**:
  - [ ] 3つのAPIが環境変数で設定可能であること
  - [ ] APIキー未設定時にエラーではなくスキップされること
  - [ ] 各ツールの検索結果が共通フォーマット（REQ-002.3）に変換されること

### REQ-010.5: コスト構造

| ツール | 無料枠 | 従量課金 | 月間想定コスト |
|--------|--------|---------|-------------|
| Brave | 2,000回/月 | $0.003/クエリ〜 | 微小（無料枠内） |

- **優先度**: P1（重要）

## REQ-011: 行政MCPデータソース連携

### REQ-011.1: 行政データソース定義

5AIパイプラインに日本の行政APIデータを統合し、リサーチ精度を向上させる。

| データソース | 提供元 | 主な内容 | API種別 |
|-----------|---------|---------|---------|
| **不動産取引価格API** | 国土交通省 | 地別・用途別の不動産取引価格 | REST API |
| **e-Stat API** | 総務省 | 人口動態、産業統計、労働統計 | REST API |
| **官公需入札API** | 中小企業庁 | 公共事業の入札情報 | REST API |

- **優先度**: P1（重要）
- **受入基準**:
  - [ ] 3つの行政データソースにアクセス可能であること
  - [ ] MCP経由での接続が可能であること
  - [ ] フォールバック構造（MCP→REST→WebSearch）が機能すること

### REQ-011.2: 案件タイプ別活用マッピング

各案件タイプで適切な行政データを活用する。

| 案件タイプ | 主な行政データソース | 利用目的 | 接続先AI |
|-----------|-------------------|---------|-----------|
| **Type A** (新規事業) | e-Stat（市場規模） | 市場規模の根拠データ | Gemini |
| **Type B** (改善) | 官公需入札（価格ベンチマーク） | 価格設定・入札価格の参考 | ChatGPT |
| **Type C** (DX) | e-Stat（技術標準） | DX推進統計・技術普及率 | Claude |
| **Type D** (危機) | 不動産価格（資産評価） | 資産価値評価・担保価値 | Grok |

- **優先度**: P1（重要）

### REQ-011.3: Phase 0統合

各AIテンプレートに行政データ取得指示を追加し、MCP接続ロジックを実装する。

**各AIの担当データソース**:

| AI | 担当行政データ | 対象案件タイプ |
|----|-------------|--------------|
| Claude | e-Stat（技術統計・DX普及率） | Type C |
| ChatGPT | 官公需入札（価格・入札情報） | Type B |
| Gemini | e-Stat（市場規模・人口動態） | Type A |
| Grok | 不動産価格（資産評価） | Type D |

**フォールバック構造**:

1. **MCP接続** → 最優先
2. **REST API** → MCP不可用時
3. **WebSearch** → API不可用時
4. **訓練データのみ** → 全不可用時（Graceful Degradation）

- **優先度**: P1（重要）
- **受入基準**:
  - [ ] 各AIテンプレートに行政データ取得指示が含まれること
  - [ ] フォールバック構造が実装されていること
  - [ ] MCP接続失敗時に自動フォールバックが機能すること

### REQ-011.4: Phase 2活用

Codexの補強分析で行政データを数値裏付けに活用する。

**活用方法**:
- Type B案件: 官公需入札データを価格ベンチマークとして使用
- Type A案件: e-Statデータを市場規模の客観的根拠として使用
- 信頼度調整: 官公需データは `reliability: "high"` で扱う

- **優先度**: P1（重要）
- **受入基準**:
  - [ ] Phase 2で行政データを数値裏付けに使用できること
  - [ ] 官公需データの信頼度が適切に反映されていること

### REQ-011.5: データ品質

行政データの品質管理ルールを定義する。

| 項目 | 規則 |
|------|------|
| **信頼度評価** | 官公需データは `reliability: "high"` で固定 |
| **更新頻度管理** | 各データソースの更新頻度をメタデータとして記録 |

  - [ ] 行政データの出典が明示されていること
  - [ ] 古さリスクが評価されていること
  - [ ] 信頼度が適切に設定されていること
