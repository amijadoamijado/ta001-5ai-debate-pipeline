# SD002開発思想・フェーズ別品質基準

## 核心思想

**90%までは高速開発、最後の10%は完璧主義**

現在のAI能力で90%までの開発は高速に実行可能。しかし、最終段階では「必ず動く」ことが唯一の成功基準。

## フェーズ別品質基準

### Phase 1-4: 高速開発フェーズ（0-90%）
**目的**: 素早く形にする、イテレーションを回す

#### 許容される行動
- ✅ 多少のミスは許容（後でコードレビューで修正）
- ✅ 効率優先の実装
- ✅ 部分的なテスト
- ✅ プロトタイプ品質
- ✅ 推測を含む実装

#### 品質基準
- コンパイルエラーなし
- 基本的な動作確認
- 主要パス動作
- カバレッジ50-70%程度

### Phase 5: 本番テスト・本番モード（90-100%）
**目的**: 「必ず動く」ことを保証する

#### 🚨 本番モード発動キーワード
ユーザーが以下のキーワードを使用した場合、即座に本番モードへ切り替え：
- **「本番」**
- **「本番テスト」**
- **「本番モード」**
- **「プロダクション」**
- **「デプロイ前」**
- **「最終確認」**

#### 本番モードでの厳格な行動規範

##### ❌ 絶対禁止事項
- ❌ **コードの飛ばし読み**: 全行を精読必須
- ❌ **部分的な確認**: 全ファイル・全関数を確認
- ❌ **推測による判断**: 必ず実際のコードで検証
- ❌ **省略**: いかなる確認工程も省略禁止
- ❌ **楽観的な想定**: 「たぶん大丈夫」は許されない
- ❌ **サンプルデータのみのテスト**: 実データ・エッジケース必須

##### ✅ 必須事項
- ✅ **完全な精読**: 全コードを1行ずつ精読
- ✅ **全ケーステスト**: 正常系・異常系・エッジケース全て
- ✅ **実環境テスト**: 本番環境と同じ条件
- ✅ **エラーハンドリング検証**: 全てのエラーケースを確認
- ✅ **依存関係の完全確認**: 外部ライブラリ・API全て
- ✅ **ロールバック準備**: 失敗時の復旧手順確認

#### 本番モードの品質ゲート

##### 1. コード完全精読
```
全ファイルを開く
↓
全関数を1行ずつ読む
↓
ロジックの完全理解
↓
潜在的バグの洗い出し
```

##### 2. 完全なテストケース
```
正常系テスト（全パターン）
↓
異常系テスト（全エラーケース）
↓
エッジケーステスト（境界値・極端値）
↓
実データテスト
```

##### 3. 環境完全一致
```
本番環境の再現
↓
依存関係の完全一致
↓
設定の完全一致
↓
データ量・負荷の再現
```

##### 4. 8段階品質ゲート全通過
```
1. 構文検証（エラー0件）
2. 型検証（警告0件）
3. リント検証（違反0件）
4. セキュリティ検証（脆弱性0件）
5. テスト検証（カバレッジ80%以上、全パス）
6. パフォーマンス検証（制限内）
7. ドキュメント検証（完全）
8. 統合検証（全シナリオ成功）
```

#### 本番モードの成功基準

**唯一の成功**: 「必ず動く」
- 全てのテストケースがパス
- 全てのエラーケースが適切にハンドリング
- 全てのエッジケースで正常動作
- 本番環境で確実に動作する保証

**失敗の定義**: 上記以外の全て
- 1つでもテストが失敗 → 失敗
- 1つでもエラーケースが未対応 → 失敗
- 推測や想定を含む → 失敗
- 「たぶん大丈夫」 → 失敗

## AIの振る舞い切り替え

### 通常開発モード（デフォルト）
- スピード優先
- 効率的な実装
- 適度な品質

### 本番モード（キーワード検知時）
```
🚨 本番モード発動

【確認事項】
1. 全コードファイルを開いていますか？ → ファイルリスト確認
2. 全関数を精読しましたか？ → 関数リスト確認
3. 全テストケースを実行しましたか？ → テスト結果確認
4. エラーハンドリングを確認しましたか？ → try-catch全確認
5. エッジケースをテストしましたか？ → 境界値テスト確認
6. 本番環境と一致していますか？ → 環境差分確認
7. ロールバック準備ができていますか？ → 復旧手順確認

【成功基準】
✅ 全ての確認事項にYES → 本番テスト実行可能
❌ 1つでもNO → 本番テスト不可、要対応
```

## 学習と適応の原則

### エラーからの学習（重要）

**基本ルール**: 同じミスを繰り返さない

#### 1回目のエラー
- ✅ 許容: 新しい問題、未知の状況
- ✅ 対応: エラー内容を記録、原因分析

#### 2回目のエラー（警告レベル）
- ⚠️ 警告: 同じパターンのエラー発生
- ✅ 必須対応:
  - より慎重なコードレビュー実施
  - トレースを詳細に実行
  - 類似エラーの可能性を洗い出し

#### 3回目のエラー（重大問題）
- 🚨 重大: 学習していない証拠
- ✅ 緊急対応:
  - **全力でシステムを動かす**
  - アプローチの根本的見直し
  - 全ファイル精読
  - 全依存関係の完全確認
  - エラーパターンの徹底分析
  - 必要ならゼロから再実装

### 臨機応変な対応

#### 許容される適応
- ✅ 状況に応じた柔軟な判断
- ✅ 効率的なアプローチへの変更
- ✅ 新しい解決策の試行

#### 許容されない適応
- ❌ 同じミスの繰り返し
- ❌ 学習しない振る舞い
- ❌ エラー後の安易な対応
- ❌ 「また同じエラー」の放置

### エラー対応の段階的強化

```
1回目エラー
└→ 通常対応: エラー分析 → 修正

2回目エラー（同じパターン）
└→ 強化対応:
   - コードレビュー徹底
   - トレース慎重実施
   - 関連エラー予測
   - テストケース追加

3回目エラー（同じパターン）
└→ 全力対応:
   - 全コード精読
   - 全依存関係確認
   - アプローチ根本見直し
   - 必要なら再実装
   - システムが動くまで全力
```

### システムを動かす全力対応

**目標**: システムが動くこと

#### 全力対応の具体例
1. **全ファイル精読**: 関連する全てのファイルを1行ずつ
2. **全依存関係確認**: ライブラリ、API、環境変数、設定全て
3. **トレース実行**: デバッガーを使った詳細トレース
4. **エッジケース洗い出し**: 全ての可能性を検証
5. **段階的検証**: 小さな単位で動作確認
6. **ログ詳細化**: 全ての実行パスをログで追跡

#### 諦めないルール
- ✅ 動くまで対応を続ける
- ✅ アプローチを変えて再挑戦
- ✅ 必要なら専門家への相談を提案
- ❌ 「動かない」で終わらない
- ❌ 安易な妥協はしない

## まとめ

- **90%まで**: 高速開発、多少のミスは許容
- **最後の10%**: 完璧主義、「必ず動く」が唯一の成功
- **本番キーワード**: 即座にモード切り替え
- **成功の定義**: 推測なし、全確認、全テスト、完璧な動作保証
- **エラー学習**: 同じミスを繰り返さない、2回目から強化対応、3回目は全力対応
- **システムを動かす**: 動くまで全力、諦めない

**重要**: このルールはSD002フレームワークの核心思想であり、全AIツールで遵守必須です。

## Ralph Loop Integration

### Purpose

Ralph Loopは「動くまで続ける」を自動化するメカニズム。
SD002の開発思想（90%高速開発・10%完璧主義）と統合し、フェーズ別に適用。

### Phase-Based Application

| Phase | Ralph Loop | Purpose |
|-------|------------|---------|
| Early (Phase 1) | Not used | Fast prototyping |
| Midpoint (Phase 2-3) | **Active** | Complete (not incomplete) |
| Endgame (Phase 4-5) | Max 2 attempts | Escalate to human collaboration |

### Midpoint Usage

高速開発でありながら「未完成」を防ぐ：

```bash
/sd002:loop-test  # Run until all tests pass
```

- max-iterations: 20
- completion-promise: ALL_TESTS_PASS

### Endgame Escalation

同一エラーパターン2回で対話型解決へ移行：

```
1st occurrence → Ralph auto-fix
2nd occurrence → Stop, escalate to /dialogue-resolution
```

### Integration with Error Learning Principle

| Error Count | Manual Response | Ralph Response |
|-------------|-----------------|----------------|
| 1st | Allowed, fix | Auto-fix loop |
| 2nd | Warning, careful | Final auto attempt |
| 3rd+ | Full effort, human | Escalate to dialogue |

### Commands

```bash
/sd002:loop-test    # Primary: test completion
/sd002:loop-lint    # Supplementary: ESLint
/sd002:loop-type    # Supplementary: TypeScript
```

### Configuration

- Stop hooks: `.claude/hooks/sd002-stop-hook*.sh`
- Rules: `.claude/rules/ralph-loop.md`
- Settings: `.claude/settings.json`
