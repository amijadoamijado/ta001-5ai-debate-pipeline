# Adapter-Core分離パターン

## 原則

> 本番データは「汚い」。システム（Core）は「綺麗」に保つ。
> 汚いデータは「玄関」（Adapter）で処理する。

## アーキテクチャ

```
       ┌─────────────────┐
       │ External Data   │ ← 本番の「汚い」データ
       └────────┬────────┘
                │
       ┌────────▼────────┐
       │  Input Adapter  │ ← 変換・正規化
       └────────┬────────┘
                │
         Standard Format  ← Interface/型定義
                │
       ┌────────▼────────┐
       │      Core       │ ← ビジネスロジック
       └────────┬────────┘
                │
         Standard Format
                │
       ┌────────▼────────┐
       │ Output Adapter  │ ← 出力変換
       └─────────────────┘
```

## 各層の責務

| 層 | 責務 | 特性 |
|----|------|------|
| Input Adapter | 外部データ → 標準形式 | 使い捨て可、データソースごとに作成 |
| Core | ビジネスロジック | 標準形式のみ受付、「汚さ」を知らない |
| Output Adapter | 標準形式 → 外部形式 | 出力先ごとに作成 |

## 開発順序（厳守）

1. **標準形式（Interface）を最初に定義**
2. Coreを標準形式前提で開発
3. Adapterは最後に接続

## ファイル配置

```
src/
├── adapters/
│   ├── input/      # 入力変換（本番データ→標準形式）
│   └── output/     # 出力変換（標準形式→外部形式）
├── core/           # ビジネスロジック（標準形式のみ）
└── interfaces/     # 標準形式の型定義（最重要）
```

## ルール

| ルール | 理由 |
|--------|------|
| Coreは外部データ形式を直接参照禁止 | 汚染防止 |
| Adapterの変更はCoreに影響しない | 独立性 |
| 標準形式を最初に定義 | 契約の明確化 |

## Env Interfaceとの関係

| パターン | 抽象化対象 | 役割 |
|---------|-----------|------|
| Env Interface | 実行環境（GAS/Local） | 環境の切り替え |
| Adapter-Core | データ品質（汚い/綺麗） | データの正規化 |

**両パターンは併用する**：
- Env Interface: 「どこで動くか」を抽象化
- Adapter-Core: 「どんなデータが来るか」を抽象化
